Giải Thích Chi Tiết Chuyên Sâu - Module Catalog (src/catalog)

Module Catalog là "quầy trưng bày" của hệ thống e-commerce, chịu trách nhiệm xử lý các luồng dữ liệu liên quan đến danh mục, sản phẩm, tìm kiếm và lọc. Đặc thù của module này là lượng truy cập đọc (read-heavy) rất lớn, do đó kiến trúc của nó tập trung tối đa vào tối ưu hoá hiệu năng (Performance) và bộ nhớ đệm (Caching).

Tài liệu này sẽ đi sâu vào sự tương tác giữa Controller và Service, cách xử lý luồng dữ liệu, và các pattern thiết kế được áp dụng.

1.2 Cơ Chế Ánh Xạ URL Query Vào DTO (@Query() query: ProductQueryDto)

@Query()
Tự động ánh xạ các tham số từ URL Query String (ví dụ: ?search=iphone) vào đối tượng DTO.
Tác động: Sử dụng class-transformer để convert kiểu dữ liệu và class-validator để kiểm tra tính hợp lệ trước khi vào Controller.

Trong endpoint GET /products, tham số @Query() query: ProductQueryDto là một minh chứng rõ ràng cho sức mạnh của NestJS.

  • Cách thức hoạt động: Dựa vào Global ValidationPipe, NestJS tự động khởi tạo (instantiate) object của class ProductQueryDto.
  • Kế thừa (Inheritance): ProductQueryDto kế thừa từ PaginationQueryDto giúp tái sử dụng logic phân trang mặc định.
  • Lợi ích: Đảm bảo controller nhận được một đối tượng an toàn, đã được xác thực kiểu dữ liệu.

2. Services: Lõi Logic Trọng Tâm (catalog.service.ts)

CatalogService là nơi thực thi toàn bộ Business Logic, sử dụng pattern tách biệt public API và private helpers.

2.1 Các Phương Thức Public (Public Methods)

  • findCategories(): Trả về danh sách danh mục, tận dụng Redis (getOrSet) để tối ưu tốc độ cho dữ liệu ít thay đổi.
  • findProducts(query: ProductQueryDto): Điều phối phân trang, tạo cache key duy nhất, và thực thi song song truy vấn Prisma.
  • findProductBySlug(slug: string): Lấy chi tiết sản phẩm qua Cache, validate trạng thái ACTIVE, ném NotFoundException nếu cần.
typescriptsrc/catalog/catalog.service.ts
async findProducts(query: ProductQueryDto) {
  const { page = 1, limit = 10 } = query;
  const skip = (page - 1) * limit;
  const cacheKey = this.getProductsCacheKey(query, page, limit);

  return this.redisService.getOrSet(cacheKey, async () => {
    const where = this.buildProductWhere(query);
    const [items, total] = await Promise.all([
      this.prisma.product.findMany({
        where, skip, take: limit,
        include: { category: true, images: true },
        orderBy: this.getProductOrderBy(query.sort)
      }),
      this.prisma.product.count({ where })
    ]);
    return { data: items.map(p => this.serializeProduct(p)), total, page, limit };
  });
}

2.2 Tương Tác Với Các Private Helpers

Việc chia nhỏ logic vào các hàm private giúp code dễ đọc và tuân thủ SRP.

  • buildProductWhere: Chuyển đổi DTO sang định dạng Prisma.ProductWhereInput.
  • getProductOrderBy: Dịch enum/chuỗi truyền từ người dùng thành object định hướng sắp xếp.
  • getProductsCacheKey: Sinh ra chuỗi định danh duy nhất cho Redis.
  • serializeProduct:
    serializeProduct
    Data Mapper đóng vai trò chuẩn hoá dữ liệu thô từ Prisma Client.
    Tác động: Biến đổi kiểu Decimal của thuộc tính price sang chuỗi string để bảo toàn độ chính xác tài chính khi trả về JSON.

3.2 Tác Động Chéo Giữa Các Module (Cross-Module Impacts)

Module Catalog phụ thuộc hoàn toàn vào trạng thái của Redis Cache. Dữ liệu mà khách hàng nhìn thấy không phải lúc nào cũng query trực tiếp từ Database.

  • Trách nhiệm làm mới dữ liệu thuộc về Admin Module.
  • Khi Admin thực hiện CUD, Admin Service sẽ invalidate các Cache Key tương ứng.

3.4 Tại Sao Lại Dùng paginated()?

Hàm paginated(data, total, page, limit) tạo ra một chuẩn phản hồi thống nhất toàn hệ thống. Nó giúp Frontend luôn biết cách đọc dữ liệu và hiển thị phân trang chính xác.

4. Tích Hợp Third-Party & Thư Viện

4.1 Tối Ưu Bằng Promise.all Với Prisma

Sử dụng Promise.all() để thực thi song song findMany và count, giảm gần một nửa thời gian chờ I/O.

4.2 Pattern Cache-Aside Với redisService.getOrSet

Cơ chế hoạt động:

  1. Kiểm tra Cache: RAM Redis có dữ liệu không?
  2. Cache Hit: Trả ngay kết quả (< 5ms).
  3. Cache Miss: Query Database -> Serialize dữ liệu -> Lưu ngược vào Redis (TTL) -> Trả về Client.

Mô hình này giúp ứng dụng "đỡ" được traffic CCU cực lớn mà không làm sập Database.
