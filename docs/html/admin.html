Giải Thích Chi Tiết - Module Admin (src/admin)

Module admin đóng vai trò là "bộ não" quản trị (Back-office) của hệ thống. Đây là nơi cung cấp các API đặc quyền để Quản trị viên (Admin) vận hành cửa hàng, bao gồm việc quản lý danh mục (Category), sản phẩm (Product) và tiến trình xử lý đơn hàng (Order).

Tóm tắt các thành phần

Thành phần Mô tả Kiến trúc Flow Liên kết
AdminController Điều phối request, bảo vệ bằng AuthGuard & RolesGuard. Guard → Ctrl → DTO Chi tiết
AdminService Xử lý logic nghiệp vụ, quản lý cache và log kiểm toán. Svc → DB → Cache Chi tiết
Lưu trữ & Trả về Quy trình đóng gói dữ liệu và tác động chéo đến cache. DTO → JSON → Client Chi tiết

1. Controller: "Người gác cổng" và Điều phối viên (admin.controller.ts)

AdminController là điểm chạm đầu tiên của mọi request gửi đến các endpoint bắt đầu bằng /admin. Trước khi dữ liệu thực sự chạm đến logic xử lý, nó phải đi qua một loạt các trạm kiểm duyệt.

1.1. Hệ thống Decorator: Bảo vệ và Phân giải dữ liệu

@Controller('admin')
Định tuyến tất cả request có prefix /admin vào class này.
@UseGuards(AuthGuard, RolesGuard)
Hai lớp khiên bảo vệ: AuthGuard kiểm tra JWT, RolesGuard kiểm tra quyền hạn.
@Roles(UserRole.ADMIN)
Chỉ cho phép tài khoản có role là ADMIN đi qua.
@Req() request
Lấy request.user.userId để ghi log hành động của Admin.
@Body() dto
Hứng và validate dữ liệu JSON (CategoryDto, ProductDto...). Trả về 400 nếu sai định dạng.
@Param('id', ParseIntPipe)
Lấy tham số id từ URL và ép kiểu sang number an toàn.
@Query() query
Trích xuất query params (phân trang, lọc trạng thái) vào DTO.

1.2. Phân tích chi tiết các Endpoint

POST /admin/categories createCategory()
Input: CreateCategoryDto
Xử lý: Kiểm tra slug duy nhất, tạo DB.
Output: Category object bọc trong success()
PATCH /admin/categories/:id updateCategory()
Input: UpdateCategoryDto
Xử lý: Kiểm tra tồn tại, cập nhật DB, dọn cache.
Output: Updated Category
POST /admin/products createProduct()
Input: CreateProductDto
Xử lý: Category check, slug check, tạo Product + Images.
Output: Product object
PATCH /admin/products/:id updateProduct()
Input: UpdateProductDto
Xử lý: Xử lý ảnh (xóa cũ tạo mới), cập nhật data, dọn cache.
Output: Updated Product
DELETE /admin/products/:id softDeleteProduct()
Input: id
Xử lý: Set deletedAt = now(), dọn cache.
Output: Success message
GET /admin/orders findOrders()
Input: AdminOrderQueryDto (page, limit, status)
Xử lý: Lấy list & count song song (Promise.all).
Output: Phân trang qua paginated()
PATCH /admin/orders/:id/status updateOrderStatus()
Input: UpdateOrderStatusDto
Xử lý: Cập nhật trạng thái, mapping DTO, ghi log kiểm toán.
Output: AdminOrderResponseDto

2. Service: "Khối óc" xử lý nghiệp vụ (admin.service.ts)

Nếu Controller là lễ tân, thì AdminService chính là những chuyên gia phía sau xử lý logic cốt lõi. Nơi đây quyết định dữ liệu có được lưu trữ hay không. Thiết kế cốt lõi của Service là việc tách biệt các luồng thao tác chính thành các hàm public và các kiểm tra nghiệp vụ (business validation) thành các hàm private (helper methods).

2.1. Chi tiết các Public Methods (Luồng xử lý chính)

Các hàm public là những API được gọi trực tiếp từ Controller, đảm nhận toàn bộ vòng đời của một tác vụ quản trị.

  • createCategory(adminUserId, dto):
    • Tạo danh mục mới.
    • Kiểm tra ensureUniqueCategorySlug để tránh trùng lặp slug.
    • Gọi Prisma tạo dữ liệu, sau đó chạy invalidateCategoryCache để xóa bộ nhớ đệm (cache) phía client và log lại hành động của Admin.
  • updateCategory(adminUserId, id, dto):
    • Cập nhật danh mục có sẵn.
    • Đầu tiên kiểm tra ensureCategoryExists, sau đó nếu có thay đổi slug, kiểm tra tiếp ensureUniqueCategorySlug (bỏ qua id hiện tại).
    • Cập nhật database, dọn dẹp cache và lưu log.
  • createProduct(adminUserId, dto):
    • Thêm một sản phẩm mới vào hệ thống.
    • Phải vượt qua các chốt chặn ensureCategoryExists và ensureUniqueProductSlug trước khi gọi database.
    • Xử lý mảng images bằng toán tử tạo liên kết (create) của Prisma.
    • Xóa cache sản phẩm theo slug và lưu log kiểm toán.
  • updateProduct(adminUserId, id, dto):
    • Cập nhật thông tin sản phẩm. DTO đưa vào có thể chỉ chứa một vài trường cần cập nhật (do tính Partial).
    • Đi qua các vòng kiểm tra: ensureProductExists, ensureCategoryExists (nếu đổi danh mục) và ensureUniqueProductSlug (nếu đổi slug).
    • Tách các trường qua toán tử spread (...rest).
    • Xử lý ảnh là một chiến thuật đặc biệt: thay vì đối chiếu tìm xem ảnh nào xóa/thêm/sửa, Service xóa toàn bộ ảnh cũ và tạo mới danh sách ảnh (deleteMany: {} tiếp theo là create).
    • Cuối cùng, tiến hành dọn cache bằng cách gọi invalidateProductCache với cả slug cũ và slug mới, rồi lưu log.
  • softDeleteProduct(adminUserId, id):
    • Xóa mềm (soft delete) sản phẩm thay vì xóa hoàn toàn khỏi database.
    • Gọi ensureProductExists để kiểm tra.
    • Cập nhật trường deletedAt thành thời gian hiện tại (new Date()).
    • Xóa cache theo slug của sản phẩm và lưu log.
  • findOrders(query):
    • Lấy danh sách đơn hàng có phân trang và lọc theo trạng thái (query.status).
    • Sử dụng Promise.all([findMany, count]) để tối ưu hóa hiệu suất.
    • Data trả về từ database được làm sạch qua lớp AdminOrderResponseDto trước khi trả về cho Controller.
  • updateOrderStatus(adminUserId, orderId, dto):
    • Cập nhật trạng thái của đơn hàng (ví dụ từ PENDING sang SHIPPED).
    • Kiểm tra sự tồn tại của đơn hàng bằng một truy vấn findUnique. Nếu không có, ném ra lỗi NotFoundException.
    • Cập nhật trạng thái mới, tự động include danh sách sản phẩm và thông tin user để định dạng lại phản hồi.
    • Trả về đối tượng đã map qua AdminOrderResponseDto và lưu log kiểm toán.

2.2. Chi tiết các Private Helpers (Hàm hỗ trợ và kiểm tra nghiệp vụ)

Các helper methods giúp tách biệt các logic kiểm tra phức tạp, làm cho các hàm public ngắn gọn, tập trung hoàn toàn vào luồng chính (Happy Path).

ensureCategoryExists(id)
NotFoundException(404) if missing
ensureProductExists(id)
NotFoundException(404) if missing
ensureUniqueCategorySlug(slug, excludeId?)
ConflictException(409) if duplicate
ensureUniqueProductSlug(slug, excludeId?)
ConflictException(409) if duplicate

invalidateProductCache(oldSlug?, newSlug?)

3.1. Hành trình chi tiết của Dữ liệu

  1. Từ DB về Service: Prisma trả về Raw Entity (chứa kiểu Decimal thô).
  2. Xử lý tại Service:
    • Mapping sang DTO (convert Decimal sang String để tránh sai số frontend).
    • Tác động chéo: Gọi Redis xóa cache (Admin sửa -> End-user thấy ngay).
  3. Đóng gói tại Controller: Dùng success() hoặc paginated() để bọc data vào cấu trúc JSON chuẩn.
  4. Về Client: JSON đồng nhất giúp Frontend dễ dàng xử lý interceptor.

4. Vai trò của Third-Party & Thư viện

  • Prisma: ORM an toàn kiểu dữ liệu, thực thi SQL.
  • Redis: Hệ thống cache. Phục vụ việc xóa cache Catalog ngay khi Admin thay đổi dữ liệu.
  • Logger: Ghi lại Audit Trail (dấu vết kiểm toán) - Ai đã sửa gì, khi nào.
