Phân Tích Chuyên Sâu Module Auth (src/auth)

Tài liệu này cung cấp một cái nhìn chuyên sâu vào cách thức hoạt động của module Xác thực (Auth). Trọng tâm là sự tương tác giữa Controller và Service, dòng chảy dữ liệu và vai trò của các thư viện bên thứ ba.

Tóm tắt các thành phần

Thành phần Mô tả Kiến trúc Flow Liên kết
AuthController Cửa ngõ giao tiếp, xử lý Serialization và Cookies. Ctrl → Interceptor → Cookie Chi tiết
AuthService Logic nghiệp vụ chính: bcrypt, JWT, và quản lý phiên. Svc → JWT → Bcrypt Chi tiết
Data Trace Luồng truyền tải Token và làm sạch dữ liệu DTO. DTO → Exclude → JSON Chi tiết

1. Controller (src/auth/auth.controller.ts)

AuthController là cửa ngõ giao tiếp, tiếp nhận yêu cầu HTTP và áp dụng các lớp bảo vệ thông qua Decorators.

1.1. Giải Mã Các Decorators Trọng Yếu

@UseInterceptors(ClassSerializerInterceptor)
Tự động hóa quá trình "làm sạch" dữ liệu nhạy cảm (như password) trước khi gửi về client.
@Res({ passthrough: true })
Cho phép thiết lập Header/Cookie mà không làm gián đoạn quy trình xử lý return của NestJS.
@UseGuards(AuthGuard)
Bảo vệ endpoint, trích xuất và verify JWT từ cookies để gắn vào request.user.

2. Service (src/auth/auth.service.ts)

AuthService là trái tim của hệ thống Auth. Nó chứa toàn bộ logic xử lý, từ việc băm mật khẩu, so sánh dữ liệu, cho đến thao tác sinh token và thiết lập cookies, bằng cách gọi qua UserService.

2.1. Các Phương Thức Public (Luồng Nghiệp Vụ Chính)

  • register(data: RegisterDto): Kiểm tra email tồn tại, băm mật khẩu người dùng, và giao cho UserService lưu vào CSDL.
  • login(data: LoginDto, response: Response): Tìm user theo email, so sánh mật khẩu qua bcrypt, sinh cặp Token mới, lưu Refresh Token vào DB và thiết lập HttpOnly Cookies.
  • handleRefresh(refreshToken, response): Xác thực Refresh Token, kiểm tra tính hợp lệ trong DB (chống Token Hijacking), sau đó cấp mới cặp Token.
  • handleLogout(accessToken, response): Xác định user, xóa Refresh Token trong DB và gỡ bỏ Cookies trên trình duyệt.
  • validateAccessToken(token): Giải mã và xác thực tính hợp lệ của Access Token, trả về payload nếu thành công.

2.2. Các Phương Thức Private (Helpers)

setCookies(response, tokens)
Cấu hình bảo mật Cookie: httpOnly: true, secure, sameSite: 'lax'.
generateTokens(userId, email, role)
Sử dụng jsonwebtoken để ký mã với Secret Key và TTL tương ứng.
updateRefreshToken(userId, token)
Băm Refresh Token bằng bcrypt trước khi yêu cầu lưu vào Database.

Hành trình chi tiết:

  1. Khởi nguồn Token: accessToken và refreshToken ra đời tại generateTokens() trong Service.
  2. HttpOnly Cookies: Service gọi setCookies() để nhét token vào Header Set-Cookie. Trình duyệt tự động lưu lại, chặn đứng tấn công XSS.
  3. Serialization: ClassSerializerInterceptor đánh chặn DTO trên đường ra, dùng class-transformer để xóa các trường @Exclude() (password, refreshToken).
  4. JSON Cuối Cùng: Client nhận được JSON sạch sẽ, an toàn cùng với trạng thái phiên được bảo mật trong Cookie.

4. Vai Trò Cốt Lõi Của Các Thư Viện

  • bcryptjs (Mã hóa một chiều):
    • Đảm bảo an toàn cho Mật khẩu và Refresh Token.
    • Salt rounds = 10: Mức cân bằng hoàn hảo giữa hiệu suất server và độ bảo mật chống Brute-force.
  • jsonwebtoken (Cơ chế Token):
    • Tạo và xác thực thẻ thông hành chuẩn RFC 7519.
    • Stateless Session: Không cần lưu giữ phiên đăng nhập ở bộ nhớ server.
    • Sử dụng Secret Keys độc lập cho Access Token (15 phút) và Refresh Token (7 ngày).
  • class-transformer & class-validator (Lá chắn dữ liệu):
    • class-validator: Phòng thủ đầu tiên cản bước các cuộc tấn công chèn mã độc ngay tại DTO.
    • class-transformer: Cung cấp @Exclude(), chốt chặn bảo mật cuối cùng ngăn chặn rò rỉ dữ liệu nhạy cảm.
