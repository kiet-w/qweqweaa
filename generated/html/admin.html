Giải Thích Chi Tiết - Module Admin (src/admin)

Module admin đóng vai trò là "bộ não" quản trị (Back-office) của hệ thống. Đây là nơi cung cấp các API đặc quyền để Quản trị viên (Admin) vận hành cửa hàng, bao gồm việc quản lý danh mục (Category), sản phẩm (Product) và tiến trình xử lý đơn hàng (Order).

Dưới đây là một phân tích chuyên sâu về cách các thành phần trong module này (Controller, Service) tương tác với nhau, luồng dữ liệu từ Client đến Database và cách trả về.

1. Controller: "Người gác cổng" và Điều phối viên (admin.controller.ts)

AdminController là điểm chạm đầu tiên của mọi request gửi đến các endpoint bắt đầu bằng /admin. Trước khi dữ liệu thực sự chạm đến logic xử lý, nó phải đi qua một loạt các trạm kiểm duyệt.

1.1. Hệ thống Decorator: Bảo vệ và Phân giải dữ liệu

Mỗi decorator trong NestJS có một nhiệm vụ cực kỳ cụ thể để đảm bảo an toàn và cấu trúc dữ liệu:

  • @Controller('admin'): Định tuyến tất cả request có prefix /admin vào class này.
  • @UseGuards(AuthGuard, RolesGuard): Hai lớp khiên bảo vệ.
    • AuthGuard: Kiểm tra JWT Token trong header Authorization. Nếu hợp lệ, nó sẽ giải mã và gắn thông tin người dùng (user payload) vào object Request.
    • RolesGuard: Chạy ngay sau AuthGuard. Nó sẽ đọc role của user và so sánh với quyền được yêu cầu.
  • @Roles(UserRole.ADMIN): Định nghĩa Metadata cho RolesGuard biết rằng: "Chỉ cho phép tài khoản có role là ADMIN đi qua".
  • @Req() request: Trích xuất toàn bộ object Request của Express. Ở đây, ta dùng nó để lấy request.user.userId (đã được AuthGuard nhét vào) nhằm mục đích ghi log (ai là người thực hiện hành động).
  • @Body() dto: Hứng dữ liệu JSON từ phần thân (body) của request, sau đó chuyển đổi và xác thực thông qua các DTO (CreateCategoryDto, UpdateProductDto...). Nếu dữ liệu sai định dạng (ví dụ thiếu name hoặc price không hợp lệ), NestJS tự động chặn lại và trả về lỗi 400 Bad Request ngay tại đây, không cho phép tiến vào Controller.
  • @Param('id', ParseIntPipe): Lấy tham số id từ URL (ví dụ /admin/products/5) và tự động ép kiểu sang dạng number an toàn thông qua ParseIntPipe. Nếu id không phải là số, sẽ trả về lỗi 400.
  • @Query() query: Trích xuất các tham số trên URL (ví dụ /admin/orders?page=2&status=PENDING) và map vào AdminOrderQueryDto.

1.2. Phân tích chi tiết các Endpoint

  • Quản lý Danh mục (POST /admin/categories, PATCH /admin/categories/:id): Nhận CreateCategoryDto hoặc UpdateCategoryDto. Chuyển dữ liệu xuống Service. Kết quả trả về được bọc qua success() - một tiện ích chuẩn hóa cấu trúc JSON ({ statusCode: 200, data: ... }).
  • Quản lý Sản phẩm (POST /admin/products, PATCH /admin/products/:id, DELETE /admin/products/:id): Nhận dữ liệu tạo/sửa hoặc id để xóa mềm. Luồng xử lý tương tự, đẩy việc nặng xuống Service và gói kết quả lại.
  • Quản lý Đơn hàng (GET /admin/orders, PATCH /admin/orders/:id/status): API GET dùng để liệt kê, gọi adminService.findOrders và sử dụng hàm paginated() để định dạng kết quả phân trang chuẩn ({ data: [], meta: { total, page, limit } }). API PATCH cập nhật trạng thái đơn hàng.

Luồng dữ liệu đến Controller: Client -> HTTP Request -> Middleware -> Guard (AuthGuard, RolesGuard) -> Pipe (ValidationPipe trên DTO, ParseIntPipe) -> Controller Method.

2. Service: "Khối óc" xử lý nghiệp vụ (admin.service.ts)

Nếu Controller là lễ tân, thì AdminService chính là những chuyên gia phía sau xử lý logic cốt lõi. Nơi đây quyết định dữ liệu có được lưu trữ hay không. Thiết kế cốt lõi của Service là việc tách biệt các luồng thao tác chính thành các hàm public và các kiểm tra nghiệp vụ (business validation) thành các hàm private (helper methods).

2.1. Chi tiết các Public Methods (Luồng xử lý chính)

Các hàm public là những API được gọi trực tiếp từ Controller, đảm nhận toàn bộ vòng đời của một tác vụ quản trị.

  • createCategory(adminUserId, dto):
    • Tạo danh mục mới.
    • Kiểm tra ensureUniqueCategorySlug để tránh trùng lặp slug.
    • Gọi Prisma tạo dữ liệu, sau đó chạy invalidateCategoryCache để xóa bộ nhớ đệm (cache) phía client và log lại hành động của Admin.
  • updateCategory(adminUserId, id, dto):
    • Cập nhật danh mục có sẵn.
    • Đầu tiên kiểm tra ensureCategoryExists, sau đó nếu có thay đổi slug, kiểm tra tiếp ensureUniqueCategorySlug (bỏ qua id hiện tại).
    • Cập nhật database, dọn dẹp cache và lưu log.
  • createProduct(adminUserId, dto):
    • Thêm một sản phẩm mới vào hệ thống.
    • Phải vượt qua các chốt chặn ensureCategoryExists và ensureUniqueProductSlug trước khi gọi database.
    • Xử lý mảng images bằng toán tử tạo liên kết (create) của Prisma.
    • Xóa cache sản phẩm theo slug và lưu log kiểm toán.
  • updateProduct(adminUserId, id, dto):
    • Cập nhật thông tin sản phẩm. DTO đưa vào có thể chỉ chứa một vài trường cần cập nhật (do tính Partial).
    • Đi qua các vòng kiểm tra: ensureProductExists, ensureCategoryExists (nếu đổi danh mục) và ensureUniqueProductSlug (nếu đổi slug).
    • Tách các trường qua toán tử spread (...rest).
    • Xử lý ảnh là một chiến thuật đặc biệt: thay vì đối chiếu tìm xem ảnh nào xóa/thêm/sửa, Service xóa toàn bộ ảnh cũ và tạo mới danh sách ảnh (deleteMany: {} tiếp theo là create). Đây là cách đồng bộ nhanh chóng, đẩy trách nhiệm quản lý danh sách ảnh về phía Frontend.
    • Cuối cùng, tiến hành dọn cache bằng cách gọi invalidateProductCache với cả slug cũ và slug mới, rồi lưu log.
  • softDeleteProduct(adminUserId, id):
    • Xóa mềm (soft delete) sản phẩm thay vì xóa hoàn toàn khỏi database.
    • Gọi ensureProductExists để kiểm tra.
    • Cập nhật trường deletedAt thành thời gian hiện tại (new Date()).
    • Xóa cache theo slug của sản phẩm và lưu log.
  • findOrders(query):
    • Lấy danh sách đơn hàng có phân trang và lọc theo trạng thái (query.status).
    • Sử dụng Promise.all([findMany, count]) để tối ưu hóa hiệu suất. Thay vì chờ lấy danh sách xong mới đếm tổng, hai query này chạy song song bằng Prisma, giảm đáng kể thời gian phản hồi.
    • Data trả về từ database được làm sạch qua lớp AdminOrderResponseDto trước khi trả về cho Controller.
  • updateOrderStatus(adminUserId, orderId, dto):
    • Cập nhật trạng thái của đơn hàng (ví dụ từ PENDING sang SHIPPED).
    • Kiểm tra sự tồn tại của đơn hàng bằng một truy vấn findUnique. Nếu không có, ném ra lỗi NotFoundException.
    • Cập nhật trạng thái mới, tự động include danh sách sản phẩm và thông tin user để định dạng lại phản hồi.
    • Trả về đối tượng đã map qua AdminOrderResponseDto và lưu log kiểm toán (chi tiết trạng thái cũ chuyển sang mới).

2.2. Chi tiết các Private Helpers (Hàm hỗ trợ và kiểm tra nghiệp vụ)

Các helper methods giúp tách biệt các logic kiểm tra phức tạp, làm cho các hàm public ngắn gọn, tập trung hoàn toàn vào luồng chính (Happy Path). Nếu có lỗi, helper sẽ chủ động ném Exception để hệ thống báo về client.

  • ensureCategoryExists(id):
    • Truy vấn database để tìm category. Nếu không tồn tại, ném NotFoundException(404) ("Danh mục không tồn tại"). Trả về đối tượng category nếu thành công.
  • ensureProductExists(id):
    • Tương tự như danh mục, tìm product trong DB. Ném lỗi NotFoundException(404) ("Sản phẩm không tồn tại") nếu ID không hợp lệ, ngược lại trả về thông tin sản phẩm.
  • ensureUniqueCategorySlug(slug, excludeId?):
    • Đảm bảo tính duy nhất của slug danh mục.
    • Nhận tham số excludeId tùy chọn (rất hữu ích khi update, bỏ qua bản ghi đang thao tác). Nếu phát hiện slug đã có trong DB, ném ra ConflictException(409) ("Slug danh mục đã tồn tại").
  • ensureUniqueProductSlug(slug, excludeId?):
    • Đảm bảo tính duy nhất của slug sản phẩm.
    • Hoạt động tương tự ensureUniqueCategorySlug, ném ConflictException(409) ("Slug sản phẩm đã tồn tại") nếu có sự trùng lặp ngoài sản phẩm đang xét.
  • invalidateCategoryCache():
    • Xóa khóa cache tập trung của danh sách danh mục (CACHE_KEYS.CATEGORIES) thông qua RedisService.
    • Đồng thời gọi reset() để dọn dẹp các cache key liên quan.
  • invalidateProductCache(oldSlug?, newSlug?):
    • Chịu trách nhiệm vô hiệu hóa bộ nhớ đệm (cache) liên quan đến sản phẩm để dữ liệu mới cập nhật có thể hiển thị cho người dùng (End-user).
    • Gọi hàm reset() tổng thể.
    • Nếu sản phẩm có slug cũ (oldSlug), xóa key cache trực tiếp dựa trên oldSlug.
    • Nếu sản phẩm cập nhật sang slug mới (newSlug và khác oldSlug), tiếp tục xóa key cache của newSlug để đảm bảo hệ thống lấy lại dữ liệu sạch nhất trong các lần truy cập sau.

3. Quản lý Kết quả trả về (Outputs & Returns)

Hành trình của dữ liệu từ khi xử lý xong ở cơ sở dữ liệu cho đến khi hiển thị trên màn hình của Quản trị viên tuân theo một luồng nghiêm ngặt và đồng nhất. Quá trình này không chỉ liên quan đến bản thân module Admin mà còn tạo ra các tác động chéo (cross-module impacts) đến các module khác trong hệ thống.

3.1. Hành trình chi tiết của Dữ liệu (Luồng Return)

  1. Từ Cơ sở dữ liệu (Prisma) về Service:
    • Sau khi thực thi các lệnh như update, create hay findMany, Prisma trả về một object nguyên thủy (Raw Entity).
    • Ở giai đoạn này, dữ liệu là các Prisma Models "thô". Nó có thể chứa các kiểu dữ liệu đặc thù của database chưa thân thiện với JSON, ví dụ như kiểu Decimal cho các trường tiền tệ (giá sản phẩm, tổng tiền đơn hàng).
  2. Xử lý bên trong Service (Mapping & Side Effects):
    • Mapping: Đối với các entities cơ bản (như Product, Category), Service có thể trả về nguyên bản. Tuy nhiên, với các đối tượng phức tạp như Order, Service sẽ chủ động tạo ra một instance mới của DTO (new AdminOrderResponseDto()). Quá trình mapping này đóng vai trò quan trọng: nó chuyển đổi các trường Decimal thành string (để tránh sai số float trên trình duyệt) và loại bỏ các trường không cần thiết.
    • Tác động chéo (Cross-module Impact): Trước khi trả dữ liệu về Controller, Service tiến hành xóa bộ nhớ đệm bằng cách gọi RedisService (ví dụ invalidateProductCache). Hành động này có tính hệ thống cao: Admin cập nhật một sản phẩm -> Cache của sản phẩm đó bị xóa -> Khi End-user truy cập vào module Catalog, CatalogService không tìm thấy cache -> Bắt buộc truy cập lại vào Database để lấy thông tin mới nhất.
  3. Từ Service về Controller:
    • AdminService hoàn tất quá trình xử lý và truyền dữ liệu (đã mapping thành DTO hoặc Raw Entity) về cho AdminController thông qua lệnh return. AdminController sẽ đón nhận kết quả này (thường qua các lời gọi await). Dữ liệu lúc này đã sẵn sàng để trình bày nhưng cấu trúc vẫn là dạng object thuần của JavaScript.
  4. Đóng gói tại Controller và trả về Client (Formatting):
    • Object từ Service không được trả trực tiếp về cho NestJS Core để gửi đi. Thay vào đó, AdminController đóng vai trò là lớp bọc cuối cùng.
    • Controller sử dụng các hàm tiện ích như success(data) (dành cho object đơn) hoặc paginated(data, meta) (dành cho danh sách phân trang) được định nghĩa tại src/common/utils/api-response.util.ts.
    • Lớp NestJS Core nhận JSON đã được đóng gói chuẩn từ Controller, tự động chuyển đổi thành HTTP Response, gán HTTP Status Code (ví dụ: 200 OK hoặc 201 Created) và truyền tải qua mạng về cho Client (Trình duyệt/App).

Tại sao phải bọc (wrapped) bằng hàm success()?

Điều này tạo ra sự đồng nhất (consistency) cho toàn bộ API. Bất kể client gọi API lấy user, đơn hàng hay sản phẩm, cấu trúc JSON trả về luôn có một định dạng duy nhất:

{
  "statusCode": 200,
  "message": "Success",
  "data": { "id": 1, "name": "..." }
}

Việc này giúp phía Frontend cực kỳ nhàn nhã: họ chỉ cần viết đúng một hàm xử lý (interceptor) chung để kiểm tra statusCode và lấy thông tin từ trường data, không phải viết lại logic parse JSON cho từng endpoint riêng biệt.

3.2. Sơ đồ Luồng Dữ liệu và Tác động Chéo (Cross-Module Impacts)

Dưới đây là sơ đồ chi tiết biểu diễn hành trình khép kín của dữ liệu, từ lúc Client gửi request cho đến lúc nhận được kết quả, cũng như tác động của nó đến module Catalog.

4. Vai trò của Third-Party & Thư viện

Module này hoạt động trơn tru nhờ sự phối hợp của ba công cụ đắc lực:

  • Prisma (PrismaService): Là cầu nối giao tiếp với cơ sở dữ liệu. Nó đóng vai trò ORM, dịch các object JavaScript (như { where: { id }, data: dto }) thành câu lệnh SQL an toàn. Prisma trả về các object có đầy đủ type-safe (an toàn kiểu dữ liệu), giúp IDE cảnh báo ngay nếu truy xuất sai tên cột.
  • Redis (RedisService): Đóng vai trò là hệ thống tăng tốc (Cache).
    • Admin là người thay đổi dữ liệu (ví dụ: đổi giá sản phẩm). Khách hàng (End-user) là người xem.
    • Nếu admin update sản phẩm nhưng bộ nhớ tạm (Cache) của khách hàng không được xóa đi, họ sẽ vẫn thấy giá cũ.
    • Do đó, sau mỗi lệnh create/update/delete thành công trên Prisma, service bắt buộc phải gọi các hàm như this.invalidateProductCache(product.slug). Redis sẽ dọn dẹp các cache key liên quan (ví dụ product:dien-thoai-iphone), buộc hệ thống phải lấy dữ liệu mới ở lần truy cập tiếp theo của khách hàng.
  • Logging (Logger): this.logger.log(...) được gọi mỗi khi có thao tác thay đổi dữ liệu.
    • Nó ghi lại chuỗi như "Admin 1 updated order 5 status PENDING -> SHIPPED".
    • Trong các hệ thống Enterprise, điều này là cốt lõi để tạo ra Audit Trail (dấu vết kiểm toán). Nếu đơn hàng bị hủy sai quy định hoặc giá bán bị thay đổi, quản trị viên cấp cao có thể dựa vào log này để biết chính xác adminUserId nào đã làm việc đó vào lúc nào.
